🔍 Query Pipeline — 10-Step RAG Flow

Every user query goes through these 10 steps inside FilteredQueryEngine.query()

1

🛡️ Intent Detection (classifier.py)

Rule-based keyword scan. Checks for 20+ trigger words: "compare", "rank", "list", "best", "vs", "both", "all", "recommend", etc. If found → BLOCKED immediately. No LLM call needed — runs in microseconds.

2

🔤 Candidate Name Extraction (matcher.py)

Tokenize query → remove 130+ stop words → create sliding windows (1-word + 2-word) → fuzzy match each window against all candidate name terms using rapidfuzz.fuzz.ratio() with 80% threshold. Handles: "pawan"→"pavanteja", "yasasvi"→"yasaswi", "trinad"→"trinadh". Returns 0 matches → BLOCKED. 2+ matches → BLOCKED. 1 match → proceed.

3

🔒 Metadata Filter Creation

Creates MetadataFilter(key="candidate_name", value="pavanteja kamma"). This scopes the upcoming vector search to ONLY this candidate's resume chunks. Without this, the search could return chunks from any resume.

4

🔍 Vector Similarity Search (LlamaIndex + ChromaDB)

index.as_retriever(filters=filters, similarity_top_k=6) → retriever.retrieve(query_str). Embeds the user query using BGE-base → searches ChromaDB for the 6 nearest vectors (filtered by candidate_name metadata) → returns chunks with similarity scores.

5

📏 Similarity Threshold Filtering

Discard any chunk with similarity score < 0.4. This removes irrelevant chunks that passed metadata filtering but are semantically unrelated to the question. If ALL chunks are below 0.4 → return "information not found".

6

🎯 Top-K Selection

Keep only the top 3 chunks after threshold filtering. This limits the context size sent to the LLM, controlling costs and preventing context window overflow.

7

📋 Context Builder

Joins the top-K chunk texts with "\n\n=== Context Chunk ===\n" separators to create the final context string.

8

📝 Prompt Builder (THE SYSTEM PROMPT)

Constructs two messages:
System: "You are an expert Technical Recruiter. Answer ONLY from retrieved context. Never hallucinate..."
User: "Retrieved Resume Context for candidate '{name}': {context}\n\nQuestion: {query}\nAnswer:"

9

🌐 OpenRouter API Call (Exactly Once)

Sends [system_msg, user_msg] to meta-llama/llama-3.1-70b-instruct via OpenRouter (openrouter.ai/api/v1) using OpenAI SDK. Temperature=0.1. Exponential backoff: 3 retries (1s, 2s, 4s). Singleton client reused across queries.

10

📤 Return Response to UI

Packages: response text, candidate_name, spelling_suggestion, is_blocked, blocked_reason, debug_info (timings, scores, prompt, raw LLM response). Displayed in Streamlit chat with candidate badge, alerts, and optional debug panel.

📐 Chunking Method

LlamaIndex SentenceSplitter

  • chunk_size = 2048 tokens
  • chunk_overlap = 0 tokens
  • Splits on sentence boundaries (not mid-word)
  • Defined in indexer.py line 59

🤔 Why These Values?

  • 2048 tokens — resumes are 1-2 pages (~500-1500 tokens). Most fit in a single chunk, preserving full context
  • 0 overlap — since resumes fit in 1-2 chunks, overlap would just duplicate content unnecessarily
  • SentenceSplitter — ensures chunks break at sentence boundaries, not mid-word or mid-phrase

🔍 Retrieval Technique

📊 Retrieval Method: Dense Vector Similarity Search

  • Type: Dense retrieval (not sparse/BM25)
  • Algorithm: Cosine similarity via ChromaDB's built-in HNSW index
  • Embedding: BAAI/bge-base-en-v1.5 (768-dim dense vectors)
  • Pre-filter: Metadata filter on candidate_name
  • Initial fetch: top-6 (similarity_top_k = top_k × 2)
  • Post-filter: Similarity threshold ≥ 0.4
  • Final selection: Top-3 chunks

🔄 Retrieval Flow

  • 1. User query is embedded using BGE-base → 768-dim vector
  • 2. ChromaDB filters collection by candidate_name metadata
  • 3. Within filtered subset, finds 6 nearest vectors (cosine similarity)
  • 4. Chunks with score < 0.4 are discarded
  • 5. Top 3 remaining chunks become the context
  • 6. Context is sent to LLM with system prompt

📌 General / Architecture Questions

What is RAG?

Retrieval-Augmented Generation. Instead of relying on the LLM's training data, we retrieve relevant documents first, then augment the prompt with that context, and then generate an answer. This ensures the LLM answers from actual data, not hallucinated knowledge.

What problem does this project solve?

Recruiters need to quickly find information about candidates from their resumes. Instead of manually reading 29+ resumes, they can ask natural language questions like "What are Ashok's Python skills?" and get instant, accurate answers grounded in the actual resume.

Why not just use ChatGPT directly?

ChatGPT doesn't have access to your private resumes. Even if you upload them, it might hallucinate or mix up candidates. Our RAG system retrieves specific resume chunks and uses a system prompt that forces the LLM to answer ONLY from those chunks — preventing hallucination.

What is LlamaIndex and why use it?

LlamaIndex is a Python framework that simplifies building RAG systems. It handles: document loading, text chunking, embedding generation, vector store integration, metadata filtering, and retrieval. Without it, we'd have to manually implement all these components.

What is the overall flow?

Two phases: Indexing (one-time: parse resumes → chunk → embed → store in ChromaDB) and Querying (per question: classify → match name → filter → search → threshold → top-K → build prompt → LLM call → display).

How many files/modules are there?

11 code files: app.py, config.py, parser.py, indexer.py, matcher.py, classifier.py, retriever.py, evaluator.py, openrouter_service.py, test_rag.py, test_retrieval.py. Plus .env, requirements.txt, and 29 resume files.

🧮 Embedding & Vector Questions

What embedding model is used?

BAAI/bge-base-en-v1.5 — a 768-dimensional dense embedding model from Beijing Academy of AI. It runs locally on CPU via HuggingFace's sentence-transformers library. No API key or cost needed.

Why BGE instead of OpenAI embeddings?

1. Free — no per-token cost. 2. Privacy — resume data stays local. 3. Fast — no network latency. 4. Quality — BGE ranks among the top models on the MTEB benchmark for retrieval tasks.

What is a vector embedding?

A vector embedding converts text into a numerical array (768 numbers). Similar texts produce similar vectors. This allows us to find relevant resume chunks by measuring the mathematical distance between the query vector and stored vectors.

What similarity metric is used?

Cosine similarity — measures the angle between two vectors. Score ranges from 0 (completely different) to 1 (identical meaning). ChromaDB uses HNSW (Hierarchical Navigable Small World) algorithm for efficient nearest-neighbor search.

What is ChromaDB?

An open-source vector database. We use PersistentClient which stores data to disk at chroma_db/. It indexes vectors using HNSW for fast approximate nearest neighbor search. It also supports metadata filtering.

What is HNSW?

Hierarchical Navigable Small World — an algorithm for approximate nearest neighbor search. It builds a multi-layer graph of vectors for fast similarity search. Much faster than brute-force comparison, with minimal accuracy loss.

✂️ Chunking Questions

What chunking strategy is used?

SentenceSplitter from LlamaIndex with chunk_size=2048 tokens and chunk_overlap=0. It splits text at sentence boundaries, ensuring chunks don't break mid-sentence.

Why 2048 chunk size?

Resumes are short documents (typically 500-1500 tokens for 1-2 pages). A 2048-token chunk means most resumes fit in a single chunk, preserving full context without fragmentation.

Why zero overlap?

Since most resumes fit in 1-2 chunks, overlap would just duplicate content. Overlap is useful for long documents where important information might span chunk boundaries — but that's rare with resumes.

What is SentenceSplitter vs TokenSplitter?

SentenceSplitter breaks at sentence boundaries (periods, newlines). TokenSplitter breaks at exact token count, potentially cutting mid-sentence. We use SentenceSplitter for cleaner, more coherent chunks.

How many chunks per resume?

Most resumes produce 1 chunk (since they're under 2048 tokens). Longer resumes may produce 2 chunks. The total ChromaDB collection contains roughly 29-40 vectors for 29 resumes.

What metadata is stored with each chunk?

Two fields: candidate_name (used for filtering) and source_file (original filename). source_file is excluded from embedding (excluded_embed_metadata_keys) so it doesn't affect similarity search.

🔍 Retrieval Questions

What retrieval technique is used?

Metadata-filtered dense vector retrieval. First, we filter by candidate_name metadata (so we only search one person's resume). Then, we do cosine similarity search within that filtered subset. This is more precise than searching all resumes.

Why metadata filtering before vector search?

Without filtering, the search might return chunks from wrong candidates. For example, if Ashok and Pawan both mention "Python", a query about "Ashok's Python skills" might return Pawan's chunks too. Metadata filtering prevents this.

Why similarity threshold of 0.4?

0.4 is a conservative threshold. Chunks with scores below 0.4 are too semantically dissimilar to the query and would add noise to the context. Empirically, relevant resume chunks score 0.5-0.9 for well-formed queries.

Why top-K = 3?

Since most resumes produce 1-2 chunks, top-3 captures all relevant chunks. More than 3 would include irrelevant chunks. Fewer than 3 might miss important context for multi-chunk resumes.

What is the difference between top_k × 2 and top_k?

We initially fetch top_k × 2 = 6 chunks, then apply the 0.4 threshold filter, then keep only top 3. This two-stage approach ensures we have enough candidates to filter from, in case some chunks score below the threshold.

Is this sparse or dense retrieval?

Dense retrieval. Sparse retrieval (BM25/TF-IDF) matches exact keywords. Dense retrieval uses semantic embeddings that understand meaning — "Python developer" and "Python programmer" match even without identical words.

🧠 LLM & Prompt Questions

What LLM is used and why?

meta-llama/llama-3.1-70b-instruct — Meta's 70-billion parameter model, instruction-tuned. We access it via OpenRouter API. Why: high quality, instruction-following capability, affordable via pay-per-token, and open-source.

What is OpenRouter?

OpenRouter is an API gateway that provides access to 100+ LLMs through a unified OpenAI-compatible API. We use it instead of running the 70B model locally (which requires expensive GPUs). It works with the standard OpenAI Python SDK.

Where is the system prompt?

In src/retriever.py, lines 229-242. It says: "You are an expert Technical Recruiter. Answer ONLY from the retrieved resume context. Never hallucinate. Never invent information." This prevents the LLM from making up information.

Why temperature = 0.1?

Low temperature makes the LLM more deterministic and factual. For a recruiter tool, we want consistent, accurate answers — not creative or varied responses. Higher temperature would introduce randomness and potential inaccuracies.

Why exactly one LLM call per query?

Each LLM call costs money and adds latency (~1-3 seconds). All filtering (intent detection, name matching, metadata filtering, threshold filtering) is done locally before making the single API call. This minimizes cost and latency.

What prevents hallucination?

Three layers: (1) System prompt explicitly says "Never hallucinate. Answer ONLY from context." (2) We only provide the specific candidate's resume chunks as context. (3) If no relevant chunks are found, we return a static message without calling the LLM.

🔤 Name Matching & Classification Questions

How does fuzzy name matching work?

Uses rapidfuzz.fuzz.ratio() — a Levenshtein-distance-based similarity score (0-100). Query tokens are compared against candidate name terms. If score ≥ 80 → match. Example: fuzz.ratio("pawan", "pavan") ≈ 90 → match!

What are phonetic variants?

Hardcoded mappings for common spelling alternatives: "pawan"→"pavan", "yasasvi"→"yasaswi", "trinad"→"trinadh", "krish"→"krishna", "shirly"→"shirley". These are added to the candidate's search term dictionary.

Why rule-based classifier instead of LLM?

Speed (microseconds vs seconds), cost (zero API calls), reliability (deterministic results). Keyword matching is sufficient for detecting comparison/ranking patterns.

What queries are blocked?

Comparisons ("Compare A and B"), rankings ("Who is the best?"), listings ("List all candidates"), aggregations ("How many candidates know Python?"), and queries with no candidate name ("What is Python?").

🧪 Testing & Evaluation Questions

How many tests are there?

40+ test cases across 6 test classes in 2 test files. Covers: parser, fuzzy matcher, classifier, full pipeline routing, app session state logic, and retrieval integration.

How to run tests?

pytest tests/test_rag.py -v (40+ tests) or pytest tests/test_retrieval.py -v (4 tests) or pytest tests/ -v (all tests).

What is LLM-as-a-Judge?

Instead of human evaluation, we use the LLM itself to score the quality of RAG outputs. Inspired by the RAGAS framework. The judge LLM receives the context + response + ground truth and outputs a score from 0.0 to 1.0.

What are the 5 evaluation metrics?

Faithfulness (grounding), Answer Relevancy (does it answer the question?), Context Precision (are chunks relevant?), Context Recall (does context cover all facts?), Answer Correctness (factual accuracy vs ground truth).

⚡ Performance & Design Questions

What is the typical query latency?

~2-5 seconds total. Breakdown: classification (~0ms), name matching (~1ms), vector search (~50-200ms), LLM generation (~1-4 seconds). The LLM call dominates latency.

How is the engine cached?

Two levels: (1) @st.cache_resource in app.py caches the FilteredQueryEngine across Streamlit reruns. (2) FilteredQueryEngine.__init__() caches ChromaDB connection, VectorStoreIndex, and OpenRouterClient.

Why singleton pattern for OpenRouterClient?

Prevents creating multiple HTTP connections. The __new__ method ensures only one instance is ever created, reused across all queries.

What happens if the API key is invalid?

The system enters Mock Mode — it still resolves candidate names and routes queries correctly, but returns "[Mock Mode] Candidate resolved to 'X'. Please configure your OPENROUTER_API_KEY." No LLM call is made.

How does retry logic work?

Exponential backoff: attempt 1 → wait 1s → attempt 2 → wait 2s → attempt 3 → fail. This handles temporary API outages or rate limits without overwhelming the server.

Why delete collection on rebuild?

Without deleting, re-indexing creates duplicate vectors (same resume indexed twice). This causes duplicate chunks in retrieval results, wasting context space and potentially confusing the LLM.
